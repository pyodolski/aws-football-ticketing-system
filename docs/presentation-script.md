### 슬라이드 5: 핵심 기술 - 오토스케일링

```
[오토스케일링 동작 과정 그래프]

정상 상태: 2개 (CPU 30-40%)
  ↓ 부하 증가
스케일 아웃: 3개 → 4개 → 5개 (CPU 60% 초과)
  ↓ 부하 감소
스케일 인: 5개 → 2개 (CPU 60% 이하, 10분 지속)

설정:
- 최소 인스턴스: 2개
- 최대 인스턴스: 5개
- 목표 CPU: 60%
```

**대본**:

> "오토스케일링은 CPU 사용률을 기준으로 작동합니다.
> 정상 상태에서는 2개의 인스턴스가 실행됩니다.
> CPU 사용률이 60%를 초과하면 자동으로 인스턴스가 추가됩니다.
> 최대 5개까지 증가할 수 있으며,
> 부하가 감소하면 다시 2개로 줄어듭니다.
> 이를 통해 비용을 절감하면서도 성능을 보장합니다."

### 슬라이드 6: 핵심 기술 - 로드밸런싱

```
[로드밸런서 동작 과정]

요청 1 → 인스턴스 A
요청 2 → 인스턴스 B
요청 3 → 인스턴스 A (라운드 로빈)

헬스 체크:
- 10초마다 /health 엔드포인트 확인
- 3회 연속 실패 시 트래픽 제외
- 자동 복구 시 트래픽 재개
```

**대본**:

> "로드밸런서는 라운드 로빈 방식으로 트래픽을 분산합니다.
> 10초마다 각 인스턴스의 상태를 확인하며,
> 장애가 발생한 인스턴스는 자동으로 제외됩니다.
> 이를 통해 무중단 서비스를 제공합니다."

### 슬라이드 7: 핵심 기술 - 데이터베이스

```
[데이터베이스 구조]

users (사용자)
  ↓ 1:N
bookings (예매)
  ↓ N:1
matches (경기)

중복 예매 방지:
- UNIQUE 제약조건 (match_id, seat_number)

```

**대본**:

> "데이터베이스는 3개의 테이블로 구성됩니다.
> 중복 예매를 방지하기 위해 세 가지 메커니즘을 사용합니다.
> 첫째, UNIQUE 제약조건으로 같은 좌석의 중복 예매를 차단합니다.
> 둘째, 트랜잭션 처리로 데이터 일관성을 보장합니다.
> 셋째, FOR UPDATE 행 잠금으로 동시성을 제어합니다."

> 둘째, 고가용성으로 무중단 서비스를 제공합니다.

## 📝 예상 질문 및 답변

### Q1: 왜 GCP를 선택했나요? AWS나 Azure와 비교하면?

**답변**:

> "GCP를 선택한 이유는 세 가지입니다.
> 첫째, 교육용 크레딧을 제공받아 비용 부담이 적었습니다.
> 둘째, Managed Instance Group과 Cloud SQL이 사용하기 쉬웠습니다.
> 셋째, 글로벌 로드밸런서가 기본으로 제공되어 설정이 간단했습니다.
> AWS나 Azure도 유사한 기능을 제공하지만, GCP가 학습 곡선이 낮았습니다."

### Q2: 오토스케일링 목표 CPU를 60%로 설정한 이유는?

**답변**:

> "60%는 성능과 비용의 균형점입니다.
> 너무 낮게 설정하면 불필요한 인스턴스가 생성되어 비용이 증가하고,
> 너무 높게 설정하면 스케일 아웃이 늦어져 성능이 저하됩니다.
> 여러 테스트 결과, 60%가 가장 적절했습니다.
> 실제 운영 환경에서는 트래픽 패턴에 따라 조정할 수 있습니다."

### Q3: 데이터베이스도 오토스케일링이 가능한가요?

**답변**:

> "Cloud SQL은 수직 확장(Vertical Scaling)만 지원합니다.
> 즉, CPU와 메모리를 증가시킬 수 있지만, 자동으로는 되지 않습니다.
> 수평 확장(Horizontal Scaling)을 위해서는 읽기 전용 복제본을 추가하거나,
> Cloud Spanner 같은 분산 데이터베이스를 사용해야 합니다.
> 현재 시스템에서는 연결 풀 관리로 충분히 처리 가능했습니다."

### Q4: 보안은 어떻게 보장하나요?

**답변**:

> "여러 계층에서 보안을 적용했습니다.
> 네트워크 레벨에서는 방화벽 규칙으로 불필요한 트래픽을 차단하고,
> 애플리케이션 레벨에서는 JWT 토큰 기반 인증과 bcrypt 비밀번호 암호화를 사용하며,
> 데이터베이스 레벨에서는 Prepared Statement로 SQL Injection을 방지합니다.
> 추가로 HTTPS, IAM 역할 기반 접근 제어도 적용했습니다."

### Q5: 실제 서비스로 운영하려면 무엇이 더 필요한가요?

**답변**:

> "세 가지가 더 필요합니다.
> 첫째, 결제 시스템 통합이 필요합니다. PG사 연동과 결제 상태 관리가 필요합니다.
> 둘째, 모니터링과 알림 시스템을 강화해야 합니다. 장애 발생 시 즉시 알림을 받아야 합니다.
> 셋째, 백업과 재해 복구 계획이 필요합니다. 데이터 손실을 방지하고 빠르게 복구해야 합니다.
> 이러한 부분들은 향후 개선 방향에 포함되어 있습니다."

### Q6: 비용이 예상보다 많이 나오면 어떻게 하나요?

**답변**:

> "GCP는 예산 알림 기능을 제공합니다.
> 특정 금액을 초과하면 이메일로 알림을 받을 수 있습니다.
> 또한, 최대 인스턴스 수를 제한하여 비용 상한선을 설정할 수 있습니다.
> 현재는 최대 5개로 제한했지만, 필요에 따라 조정 가능합니다.
> 예약 인스턴스를 사용하면 최대 55%까지 비용을 절감할 수 있습니다."
